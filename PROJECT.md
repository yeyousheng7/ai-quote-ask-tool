# ChatGPT Quote Annotation Plugin - 产品约束文档

## 1. 产品目标

本插件用于增强 ChatGPT 网页端的“引用提问”体验。

目标不是替代 ChatGPT 官方引用功能，而是在官方引用功能旁边提供一个轻量的“批注式引用提问”能力。

用户可以对某一段 ChatGPT 回复文字进行划选，然后选择“批注提问”，插件会为该段引用创建一个独立的小窗讨论线程。之后围绕同一段引用的多次追问，都应统一保存在同一个小窗中。

整体体验参考 Word 批注：
正文中保留轻量标记，小窗中保存围绕该引用的提问和回答。

---

## 2. 核心原则

### 2.1 不破坏官方功能

插件不应破坏 ChatGPT 原生功能。

官方引用按钮应尽量保留原状。
插件可以在官方引用按钮旁边新增一个“批注提问”入口，但不建议直接重写官方按钮行为。

推荐交互：

- 官方“引用”按钮：走 ChatGPT 原生引用流程；
- 插件“批注”按钮：走本插件的小窗引用流程。

---

### 2.2 一个引用对应一个讨论线程

每一个被选中的引用片段，都对应一个独立的 quote thread。

同一个引用被多次追问时，不应创建多个窗口，而应继续追加到该引用已有的小窗线程中。

---

### 2.3 宁可少做智能合并，也不要错误合并

不要仅凭引用文字是否相同来判断两个引用是否相同。

不同位置的相同文字，默认视为不同引用。
重叠引用默认也视为不同引用。

错误合并会严重破坏用户体验，因此默认策略应保守。

---

### 2.4 自用 MVP 优先

当前插件主要面向自用和 vibecoding，不追求完整商业级能力。

第一版应优先保证：

- 能选中文字；
- 能创建引用标记；
- 能打开小窗；
- 能在小窗中继续追问；
- 能将同一引用的问答归档到同一线程；
- 能在页面刷新或切换旧回复时尽量恢复引用。

---

## 3. 基础用户流程

### 3.1 创建引用

用户在某一条 ChatGPT 回复中划选一段文字。

划选后，插件在官方浮动菜单附近新增入口，例如：

```text
[引用] [批注]
```

其中：

* `[引用]`：保留官方行为；
* `[批注]`：创建插件引用 thread。

点击 `[批注]` 后：

1. 插件记录被选中的引用内容；
2. 在正文附近生成引用标记；
3. 打开右侧或悬浮小窗；
4. 用户可以在小窗中输入问题。

---

### 3.2 同一引用继续提问

用户再次点击同一个引用标记时，应打开同一个小窗 thread。

小窗中显示该引用相关的历史问答，例如：

```text
你：为什么这里推荐 Tampermonkey？
ChatGPT：因为它适合做网页增强……

你：如果官方 DOM 变了怎么办？
ChatGPT：可以通过更保守的选择器和 MutationObserver 降低风险……
```

底部应有输入框：

```text
继续追问这个引用……
```

用户在这里提问时，插件应自动带上该引用内容作为上下文。

---

## 4. 引用标记规则

### 4.1 引用标记样式

正文中的引用可以使用轻量标记，例如：

```text
被引用的文字 [引用 1]
```

或者：

```text
被引用的文字 引用¹
```

引用文字本身可以使用：

* 浅色背景；
* 虚线下划线；
* 细边框；
* 轻量高亮。

不要使用过重的颜色，避免干扰正文阅读。

---

### 4.2 有追问时的标记

如果某个引用 thread 中已经有多次用户提问，可以在按钮中显示数量。

推荐格式：

```text
引用 1
引用 1 · 2
引用 1 · 3
```

其中数字表示该引用下的用户提问数量，而不是总消息数量。

---

### 4.3 当前激活状态

当某个引用的小窗正在打开时，对应引用标记应有轻微高亮。

可以使用：

* 更深的边框；
* 更明显的背景；
* 与小窗之间的连接线；
* 小窗标题中显示对应编号。

---

### 4.4 未读状态

如果某个引用 thread 中有新回复但用户尚未查看，可以给引用标记加小圆点。

例如：

```text
引用 2 · 1 ●
```

该功能不是 MVP 必需项，可以后续实现。

---

## 5. 引用唯一性规则

### 5.1 不按文字内容合并

以下情况即使文字相同，也默认视为不同引用：

* 来自不同 ChatGPT 回复；
* 来自同一回复但不同位置；
* 一个引用包含另一个引用；
* 两个引用部分重叠。

---

### 5.2 建议的引用身份信息

实现上可以参考以下信息判断引用身份，但不要求必须完全照此实现：

```ts
type QuoteAnchor = {
  quoteId: string
  sourceConversationId: string
  sourceMessageId: string
  sourceVariantId?: string
  startOffset: number
  endOffset: number
  exactText: string
  prefixText?: string
  suffixText?: string
  threadId: string
}
```

其中：

* `sourceConversationId`：当前 ChatGPT 会话；
* `sourceMessageId`：来源消息；
* `sourceVariantId`：同一消息的不同重新生成版本，可选；
* `startOffset / endOffset`：引用在原文中的位置；
* `exactText`：引用的精确文本；
* `prefixText / suffixText`：引用前后少量上下文，用于恢复定位；
* `threadId`：该引用对应的小窗讨论线程。

---

## 6. 重叠引用处理

重叠引用是允许的。

例如：

```text
原文：ABCDEFG
引用 1：ABCDE
引用 2：CDEFG
重叠部分：CDE
```

这种情况下：

* 引用 1 和引用 2 是两个独立引用；
* 不自动合并 thread；
* 重叠区域可以共享视觉高亮；
* 点击重叠区域时，可以弹出选择菜单。

推荐菜单：

```text
此处包含多个引用：

- 引用 1：ABCDE...
- 引用 2：CDEFG...
```

用户选择哪个引用，就打开哪个引用的小窗。

---

## 7. 跨段引用处理

第一版可以允许跨段引用，但必须满足一个限制：

> 引用不能跨越不同的 ChatGPT 回复。

允许：

* 同一条回复内跨多个段落；
* 同一条回复内普通文本 + 代码块；
* 同一条回复内普通文本 + 公式文本。

不允许：

* 从一条 ChatGPT 回复跨到另一条 ChatGPT 回复；
* 从用户消息跨到 ChatGPT 回复；
* 从页面其他区域跨到 ChatGPT 回复。

如果用户选择范围跨越多个回复，应提示：

```text
暂不支持跨回复引用，请重新选择同一条回复中的内容。
```

---

## 8. 小窗行为约束

### 8.1 小窗内容

小窗只显示该引用相关的问答 thread。

不要在小窗中塞入太多额外信息。

推荐结构：

```text
引用 1

你：……
ChatGPT：……

你：……
ChatGPT：……

[继续追问这个引用……] [发送]
```

可以在标题区域显示一个很短的引用预览，但不应占用太多空间。

---

### 8.2 同一时间打开窗口数量

第一版建议只允许同时打开一个小窗。

用户点击不同引用时，右侧小窗切换到对应引用 thread。

不要第一版就支持多个小窗并排、拖拽、固定等复杂交互。

---

### 8.3 小窗关闭

关闭小窗不应删除 thread。

关闭后，再点击对应引用标记，应恢复显示原 thread。

---

### 8.4 小窗输入

小窗底部输入框用于继续围绕当前引用提问。

输入框 placeholder 推荐：

```text
继续追问这个引用……
```

发送后，问题应追加到当前引用 thread，而不是创建新 thread。

---

## 9. 小窗提问的上下文规则

### 9.1 自动附加引用内容

当用户在小窗中继续提问时，插件应自动把引用内容加入上下文。

用户只需要输入自己的问题，不需要手动复制引用文字。

用户看到的是：

```text
你：这里为什么推荐 Tampermonkey？
```

内部可构造类似上下文：

```text
你正在回答用户围绕某段引用的追问。

<quote>
这个需求最适合自己写一个 Tampermonkey 脚本。
</quote>

<user_question>
这里为什么推荐 Tampermonkey？
</user_question>
```

具体 prompt 格式由实现者决定，不强制。

---

### 9.2 显示内容和发送内容分离

插件应区分：

* 用户真正输入的问题；
* 插件自动附加的引用上下文；
* 用于显示的小窗消息；
* 用于发送给 ChatGPT 的内部 prompt。

不要把内部 prompt 原样显示在小窗中。

---

### 9.3 主聊天折叠显示

如果底层仍然通过 ChatGPT 主输入框发送消息，那么主聊天中可能会出现内部 prompt。

为避免污染主聊天体验，建议将这类消息折叠显示为：

```text
围绕 引用 1 的批注提问
已收纳到右侧小窗
[展开查看]
```

不要完全隐藏，因为用户需要知道这次提问确实发生过。

---

## 10. 流式回复处理

第一版不要求实现小窗内流式同步。

推荐策略：

1. 用户在小窗发送问题；
2. 小窗立即显示用户问题；
3. ChatGPT 回复生成期间，小窗显示：

```text
ChatGPT：生成中……
```

4. 主聊天回复完成后，将最终回复内容归档到小窗；
5. 如果用户停止生成，则保存当前内容，并标记：

```text
回答未完成
```

后续可以再优化为实时同步流式内容，但不是第一版必需功能。

---

## 11. 重新生成与切换回复版本

ChatGPT 回复可能被重新生成，也可能在多个版本之间切换。

第一版规则：

### 11.1 重新生成后

如果当前显示的回复内容变了，原引用无法匹配，则引用标记直接消失。

不要强行把旧引用迁移到新回复上。

---

### 11.2 切回旧回复后

如果用户切回之前的回复版本，并且插件能重新匹配到原引用内容，则恢复引用标记和对应 thread。

恢复时可依赖：

* `exactText`
* `prefixText`
* `suffixText`
* `startOffset / endOffset`
* 当前消息内容匹配结果

只要能稳定匹配，就恢复；匹配不稳定则不显示。

---

## 12. 代码块处理

第一版可以将代码块作为普通文本处理。

推荐显示策略：

* 小窗内可以用简单 `pre + code` 渲染；
* 不强求完全复刻 ChatGPT 官方代码块样式；
* 保留换行和缩进；
* 超长代码允许横向滚动；
* 可选增加复制按钮。

不要第一版就追求和官方代码块完全一致。

---

## 13. 公式处理

第一版公式可以降级为纯文本显示。

如果后续要增强公式体验，推荐方向：

* 优先保存原始 Markdown / LaTeX；
* 使用 KaTeX 渲染基础公式；
* 无法获得原始公式时，保留纯文本；
* 不要求实现复杂公式编辑或复制。

公式不是第一版核心功能，不应阻塞 MVP。

---

## 14. 数据存储

### 14.1 本地存储

第一版可以使用浏览器本地存储方案，例如：

* `localStorage`
* `IndexedDB`

数据应至少按 conversation id 隔离。

---

### 14.2 建议保存的数据

每个引用 thread 可保存：

```ts
type QuoteThread = {
  threadId: string
  quoteId: string
  quoteText: string
  sourceConversationId: string
  sourceMessageId: string
  sourceVariantId?: string
  anchor: QuoteAnchor
  messages: QuoteThreadMessage[]
  createdAt: number
  updatedAt: number
}
```

其中：

```ts
type QuoteThreadMessage = {
  role: "user" | "assistant"
  content: string
  createdAt: number
  status?: "generating" | "completed" | "interrupted" | "failed"
}
```

以上仅为参考结构，不强制实现。

---

### 14.3 清理数据

应提供至少一种清理方式，例如：

* 清除当前会话的引用记录；
* 清除所有引用记录；
* 删除某个引用 thread。

自用版本可以简化，但不要完全没有清理能力。

---

## 15. 隐私与安全

插件默认只应在本地保存数据。

除非用户明确配置 API 或远程同步，否则不要把以下内容发送到第三方服务器：

* 用户聊天内容；
* 引用内容；
* 小窗 thread；
* 当前会话信息。

如果未来发布给他人使用，应在说明中明确：

* 数据保存位置；
* 是否上传数据；
* 如何清除数据。

---

## 16. DOM 与兼容性约束

ChatGPT 网页结构可能经常变化，因此插件实现不应过度依赖脆弱 className。

建议：

* 使用 MutationObserver 监听页面变化；
* 尽量基于消息结构、文本内容和可访问性属性定位；
* 避免大量写死动态 class；
* 出错时优雅降级，不影响 ChatGPT 原本使用。

---

## 17. 第一版明确不做的功能

为了控制复杂度，第一版暂不做：

* 多个小窗同时打开；
* 引用 thread 自动合并；
* 跨多个回复引用；
* 复杂公式渲染；
* 完整 Markdown 编辑器；
* 云端同步；
* 多设备同步；
* 移动端适配；
* 拖拽式窗口管理；
* 引用关系图谱；
* 完整商业化设置面板。

---

## 18. MVP 验收标准

第一版完成后，应至少满足：

1. 用户可以在同一条 ChatGPT 回复中划选文字；
2. 浮动菜单中出现插件的“批注”入口；
3. 点击“批注”后，正文中出现引用标记；
4. 点击引用标记，可以打开对应小窗；
5. 小窗可以输入问题；
6. 同一个引用的多次追问会进入同一个 thread；
7. 不同引用，包括重叠引用，不会被错误合并；
8. 主聊天中的内部 prompt 不直接污染阅读体验，至少可以折叠；
9. ChatGPT 生成期间，小窗显示“生成中”；
10. 生成完成后，小窗中保存用户问题和 ChatGPT 回复；
11. 页面刷新或切回旧回复后，能尽量恢复已存在的引用；
12. 引用无法匹配时，不强行显示错误引用；
13. 插件异常时，不影响 ChatGPT 原有功能。

---

## 19. 推荐开发优先级

### P0：必须完成

* 划选文字；
* 新增“批注”入口；
* 创建引用标记；
* 打开小窗；
* 小窗继续提问；
* 自动附加引用上下文；
* 同一引用 thread 归档；
* 基础本地存储。

### P1：体验增强

* 主聊天内部 prompt 折叠；
* 页面刷新后恢复引用；
* 切换旧回复后恢复引用；
* 重叠引用选择菜单；
* 生成中状态；
* 回答未完成状态。

### P2：后续优化

* 代码块更好渲染；
* KaTeX 公式渲染；
* 未读提示；
* 引用搜索；
* thread 删除 / 重命名；
* 设置面板；
* 导出引用讨论记录。

---

## 20. 最终产品定位

本插件应保持轻量。

它不是一个完整聊天客户端，也不是知识库系统。
它只是给 ChatGPT 网页端增加一种更自然的“围绕局部文本继续讨论”的交互。

核心体验可以概括为：

> 正文中轻量标记，小窗中持续追问，主聊天保持干净。
